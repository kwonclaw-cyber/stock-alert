    📊 분석 보드 — 변경이 매출에 준 직접 영향

    각 카드의 큰 숫자 = 순효과(%p) = 그날 매출(전주 동일요일 대비) − 기준선. 변경만의 영향에 가깝게 보정한 값입니다. ＋면 매출 견인, －면 매출 하락.

    해석 주의: "변경한 날 하루" 비교는 사장님 반응(매출 부진 → 광고·할인 ON)이 섞여 역인과일 수 있습니다. 지속적 변경(운영시간 등)과 아래 추이 그래프·구간 비교를 함께 보세요.

    📈 매출 추이 & 변경 시점

    일매출과 7일 이동평균선으로 흐름을, 세로선(변경 시점)으로 전·후를 비교합니다.

    월별 비교

    🔎 변경 → 매출효과 도출

    각 변경 직전/직후를 동일 길이·동일 요일구성(주 단위, 최대 2주) 창으로 비교해 그 변경이 매출에 준 효과를 도출합니다. 변경은 카테고리별 하루 1건(그날 순변화)으로 합쳐, 운영시간은 그날 실제 운영시간·입찰은 그날 순입찰변동·할인은 그날 순시작/종료만 셉니다. 변화 없음·측정 불가(전·후 1주 미확보)는 제외합니다. ※ 변경이 잦은 구간은 창 안에 다른 변경이 섞일 수 있어, 단독 인과가 아닌 참고치입니다.

    ⚠️ 이 페이지는 엑셀 처리를 위해 인터넷에서 라이브러리(SheetJS)를 한 번 불러옵니다. 인터넷이 연결돼 있어야 합니다.

    표본매장 분석 생성기 · 모든 처리는 브라우저 내에서 수행되어 데이터가 외부로 전송되지 않습니다.

    🆚 기준매장 vs 선택매장 1:1 비교

    잘 되는 기준매장과 개선 대상 선택매장의 파일(JSON+엑셀)을 각각 올리면, 전략·매출을 비교하고 선택매장 솔루션을 제안합니다.

        🧭 표본매장 프로젝트 — 진행 기록 & 로직 정리
        
  
        처음 접근부터 시행착오·각 단계 결과·최종 결론까지 한 페이지로
        

  
  
        0 · 목표
        
  
        
    
        브랜드별 표본매장을 정하고, 그 매장의 배민/쿠팡 운영 변경(광고율·즉시할인·쿠폰·운영시간 등)이
    매출 변동과 어떤 관계인지 분석할 수 있는 표본 데이터를 만든다.
        
  
        

  
  
        1 · 요구사항 확정 (질문으로 좁힘)
        
  
        
    
        질문결정
        
    
        운영변경 데이터 소스배민/쿠팡 사장님 사이트에서 수집
        
    
        표본매장 선정브랜드별 수기 지정
        
    
        계정 구조매장별 개별계정
        
    
        작업 성격일회성, 로그인은 사용자가 직접
        
    
        핵심 데이터현재값이 아니라 “변경 이력”(시간축)
        
    
        결과물매출과 결합한 표(엑셀/시트)
        
  
        
  
        방향 전환 포인트: 처음엔 “현재 설정값”을 받으려 했지만, 실제로 필요한 건 “언제 무엇을 어떻게 바꿨나”였다. 그리고 배민에 정확히 그걸 주는 변경이력 화면이 있었다.
        

  
  
        2 · 시도와 결과 (로직이 진화한 과정)
        
  
        

    
        
      
        시도 1 화면 저장 방식 capture.py
        
      
        로그인 후 변경이력 페이지의 HTML·스크린샷·네트워크 JSON을 통째로 저장.
        
      
        결과: 성공. 96개 응답 캡처 → 99건 파싱.
        
      
        핵심 발견: 배민 변경이력은 깔끔한 JSON API(modify-history/shop · instant-discount · ad-campaign)이고,
      변경 전→후 값이 그대로 들어있다.
        
      
        한계: 수동 클릭 의존, 페이지네이션이 중간에 잘려 6개월 완전판이 안 됨.
        
    
        

    
        
      
        시도 2 자동수집기 baemin_history.py (여러 버전)
        
      
        “로그인만 하면 6개월치를 자동으로” 목표로 Playwright 자동화 반복 시도.
        
      
        
        
        방식결과
        
        
        외부에서 API 호출(ctx.request)403 (인증 토큰 없음)
        
        
        페이지 안 fetch(page.evaluate)CORS / 실행컨텍스트 파괴(-1)
        
        
        루트 시작 + 로그인 가드로그인 창이 추적 밖으로 이탈
        
        
        실제 Chrome(persistent)로그인된 창을 Playwright가 못 잡음
        
      
        
      
        결론: Playwright로는 막힘. 배민 SSO 로그인이 자동화가 제어 못 하는 별도 창으로 빠진다.
        
      
        이때 배운 진짜 로직: 자동화의 벽은 데이터가 아니라
      “인증된 세션을 바깥에서 건드릴 수 없다”(SSO·2FA·봇차단)는 점이었다.
        
    
        

    
        
      
        전환 · 정답 개발자도구 콘솔 스니펫 baemin_console.js
        
      
        발상 전환: 우리가 요청을 만들지 말고, 로그인된 페이지가 스스로 부르는 응답을 가로채자.
        
      
          
        
        • 토큰 실어 replay → 여전히 토큰 못잡음·403·CORS
          • 
        
        • 최종 “응답 가로채기” → 토큰·CORS·403 문제가 원천적으로 사라짐
          • 
      
        
      
        성공! 밥구정로데오(14323681) 6개월 변경이력 190건 수집
      (가게 80 · 광고 84 · 즉시할인 26).
        
    
        

    
        
      
        결합 매출 합치기 merge_sales.py
        
      
        matetech 월별 엑셀(주문경로별=배민 일별매출, 시간대별)을 6개월 합본하고,
      변경이력과 날짜로 조인.
        
      
        결과: 일자별 타임라인 169일 + 시간대별 2,033행 + 변경이력 190건 통합 워크북.

        1. 운영변경은 배민, 매출은 matetech — 두 출처를 날짜로 잇는다.

          3.

          4. 
      
        4. 사람은 로그인 + 탭 조회만, 나머지(누적·정렬·표 변환)는 코드가.
